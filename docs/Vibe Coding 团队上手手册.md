# Vibe Coding 团队上手手册

> 适用项目：账务核心系统
> 适用对象：全体开发成员（含非 Java 背景同学）
> 更新维护：`docs/prompt/` 仓库负责人

---

## 写在前面：重新定义开发者的工作

斯坦福大学最近开了一门专门讲 Vibe Coding 的课程，里面有一个数字让人印象深刻：

> **40% 做规划 → 20% 写代码 → 40% 测试与验证**

这和传统开发的时间分配完全相反。传统模式下，写代码占据了大部分时间；Vibe Coding 模式下，**写代码反而是最轻的环节，AI 来做**。人的精力应该转移到两端：

- **前端**：把需求定义清楚，把上下文准备好，把边界说明白
- **后端**：验证 AI 写的是否正确，测试覆盖是否足够，逻辑是否符合预期

换句话说，程序员的核心竞争力从「能写代码」变成了「能定义问题 + 能判断对错」。

**本手册就是帮你建立这套新工作方式。**

## 第一章：大语言模型的工作原理（实用版）

> 不讲模型架构，只讲「为什么 AI 会这样工作」，以及这些原理如何直接影响你写 Prompt 的方式。

---

### 1.1 AI 怎么「理解」你的话：Token 与上下文窗口

**Token 是什么**

AI 不是按字或词来处理文本的，而是按「Token」。Token 大致可以理解为：

- 英文：一个单词约等于 1~2 个 Token
- 中文：一个汉字约等于 1~2 个 Token
- 代码：一行约等于 5~20 个 Token

**上下文窗口是什么**

AI 在每次回答时，能「看到」的内容是有上限的，这个上限叫上下文窗口（Context Window）。

```
┌─────────────────────────────────────────┐
│              上下文窗口（有上限）          │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │ 规范层   │  │ 对话历史 │  │当前Prompt│  │
│  │（自动注入）│  │（越来越长）│  │         │  │
│  └─────────┘  └─────────┘  └─────────┘  │
│          三者总量不能超过窗口上限           │
└─────────────────────────────────────────┘
```

**这对你意味着什么：**

- 一次对话越长，窗口里装的「对话历史」越多，留给当前 Prompt 的空间越少
- 当窗口被历史对话塞满时，AI 开始「遗忘」早期的约束和规范
- 这就是为什么长对话后 AI 会开始违反你最初设定的规则

**实用结论**：**会话要短，上下文要密。** 超过 10 轮就考虑开新会话。

---

### 1.2 AI 为什么会「遗忘」：注意力的稀释

AI 处理上下文时，并非对每个位置的信息一视同仁。越靠近当前 Prompt 的内容，AI 越关注；越靠前的内容，影响力越弱。

这解释了一个常见现象：

> 你在对话开头说「金额必须用 BigDecimal」，写了几轮之后，AI 又开始用 `double` 了。

不是 AI「故意」违反，而是这条规定在上下文里的位置越来越靠前，注意力权重自然降低。

**实用结论：**
- 重要约束放在规范层（自动注入到每次对话的起始位置）
- 单次 Prompt 里，最关键的约束放在**末尾**，比放在开头效果更好
- 「不要做」的禁止项比「要做」的正向要求更不容易被遗忘

---

### 1.3 AI 为什么会「自由发挥」：概率补全的本质

大语言模型的本质是**根据已有内容预测下一个 Token 的概率**。它不是在「理解」需求然后「思考」方案，而是在不断预测：

> 给定前面所有这些 Token，下一个最可能出现什么？

这意味着：

- Prompt 越模糊 → 可能的补全方向越多 → AI 越容易「发挥」
- Prompt 越具体 → 可能的补全方向越少 → 输出越可控
- 训练数据里「写代码」的常见模式会被高概率复现（比如：写个查询顺手加个缓存）

**实用结论：**
- 模糊 Prompt → 高方差输出（结果不可预期，可能加很多你不想要的东西）
- 精确 Prompt → 低方差输出（稳定可预期）
- 「不要做」的否定句，本质是在压低某些 Token 序列出现的概率

---

### 1.4 温度参数（Temperature）

AI 工具通常有一个「温度」参数，控制输出的随机性：

| 温度 | 特点 | 适合场景 |
|------|------|---------|
| 低（接近 0） | 确定性强，每次输出几乎相同 | 代码生成、格式化输出 |
| 中（0.3~0.7） | 有一定变化，但保持连贯 | 文档编写、方案设计 |
| 高（接近 1） | 创意性强，但可能不稳定 | 头脑风暴、命名建议 |

大多数 AI 编码工具的默认温度已针对代码生成做了优化，**一般不需要手动调整**。

了解这个参数的意义在于：如果 AI 每次生成的代码差异很大，可能是工具的温度偏高，或者你的 Prompt 信息量不足（AI 有太多「发挥空间」）。

---

### 1.5 System Prompt 与 User Prompt 的区别

AI 对话实际上有两个输入层：

```
System Prompt（系统层）
├── 定义 AI 的角色、行为规范、全局约束
├── 对整个会话生效
└── 对应我们的：规范层（自动注入的规范文本）

User Prompt（用户层）
├── 每次对话的具体任务
├── 只对当前轮次生效
└── 对应我们的：四段式 Prompt
```

**实用结论**：不要把规范重复写在每次的 User Prompt 里，应该放在 System Prompt 层（即规范层）。规范层写一次，永久生效；User Prompt 只写当前任务。

## 第二章：四个核心概念

> 把 AI 编程助手想象成刚入职的实习生，四个概念对应他工作的四个维度。

---

### 2.1 Prompt｜每次交代的任务

**一句话**：说一次，做一次，说完就忘。

每次开新对话，AI 不记得上次你说了什么，它只能看到当前窗口里的内容。

**最常见的错误：**

```
❌ "帮我写一个记账接口"

✅ "你是一名 Java 后端工程师，技术栈 Spring Boot 3.x + MyBatis-Plus + MySQL 8.0。
   请实现记账入口接口的入参校验逻辑，要求：
   1. 接收参数：traceNo（幂等键）、amount（BigDecimal）、businessCode
   2. 金额必须大于 0，否则抛 ServiceException(ResultCode.INVALID_AMOUNT)
   3. businessCode 不能为空，否则抛 ServiceException(ResultCode.INVALID_PARAM)
   4. 只返回校验结果 Result<Void>，不要做任何数据库操作
   5. 不要加缓存，不要加日志"
```

```
❌ 错误示范
"帮我分析下这个方案"

✅ 正确示范  
请你以国际顶尖企业智能体(Enterprise AI Agent)领域的专家视角,
结合 David Luan(Adept)、Dario Amodei(Anthropic)、Harrison Chase(LangChain)
三位顶级专家的方法论与工程实践,
对我提供的企业智能体方案进行**全面专业评估、风险识别、架构优化**。
要求:
1. 保持专家严谨性,从企业级落地、安全性、可扩展性、性能、体验五个维度评估。
2. 严格遵循我方案中的核心架构:Router → Skill → MCP → 记忆机制 → RAG → 权限体系。
3. 优化必须**可落地、可工程化、可直接开发**,不做空泛建议。
4. 最终输出一份**完整、规范、可直接使用的企业智能体技术方案文档**,
   格式为纯原始 Markdown,不含渲染,不含代码,流程图使用 mermaid。
5. 文档必须包含:需求说明、功能说明、约束说明、流程说明、系统对接、架构图、优化总结。
现在,请开始评估并输出完整方案（markdown 格式）。
```

差距在哪？**角色、技术栈、输出边界、禁止行为**——这四个要素对应第一章讲的「降低输出方差」。

---

### 2.2 Rules / Skills｜行为准则

**一句话**：写一次，永久生效，不用反复说。

把规范文本配置到 AI 工具里，工具会在每次对话时自动把它附加到 System Prompt 层。

**没有规范层的痛：**
- 今天告诉 AI "金额用 BigDecimal"，明天它又用 `double` 了
- 今天告诉 AI "异常用 ServiceException"，下次又造了一个 `BizException`
- 每次开新对话都要重新贴一遍规范，费时还容易漏

**有了规范层之后：**
规范只写一次，后续所有 Prompt 自动继承。AI 就像签了入职合同的实习生，规范是他的行为准则手册。

---

### 2.3 Projects｜工位和文件柜

**一句话**：放着项目所有背景资料，AI 随时能翻。

大多数 AI 工具支持「项目」概念：你可以上传文档，AI 在每次对话时都能访问，不需要手动粘贴。

**不用 Projects 的痛：**
- 每次都要自己粘贴表结构
- AI 生成的代码和项目已有风格对不上
- 每次都要重新解释系统架构背景

**我们项目放入 Projects 的内容：**

```
📁 项目背景文档（放入 Projects）
├── 领域模型速查表（25 张核心表的关键字段 + 关联关系）
├── 核心状态机（凭证 / 分录 / 账户 / 事务状态流转）
├── 主流程泳道图（标准入账五阶段 + 缓冲路径 + 日切流程）
└── 技术栈约定（版本号 + 核心依赖 + 禁用清单）
```

> **注意**：Projects 不是越多越好。上下文窗口是有限的，放太多会稀释 AI 的注意力。
> 原则：**只放在任何任务中都可能需要查的内容**，单次任务的细节用 Prompt 传。

---

### 2.4 MCP｜工牌上的门禁权限

**一句话**：让 AI 从「说话」变成「动手」。

MCP（Model Context Protocol）是一个协议，让 AI 能够调用外部工具和服务。

```
没有 MCP：AI 生成代码  →  你自己去执行
有了 MCP：AI 直接调用接口、查数据库、触发操作
```

**类比：**
- Prompt = 告诉实习生"帮我查一下这个账户的余额"
- MCP = 实习生真的有数据库账号，可以直接登录去查，并把结果告诉你

**我们项目规划的 MCP Tools（Phase 7 实现）：**

| Tool 名称 | 功能 | 权限级别 |
|-----------|------|---------|
| `query_account_balance` | 查账户余额（含可用 / 冻结 / 预估） | 开放 |
| `get_voucher_by_no` | 按凭证号查凭证详情 | 开放 |
| `check_trial_balance` | 查最近一次试算平衡结果 | 开放 |
| `post_accounting_entry` | 发起记账 | **需人工确认** |
| `trigger_freeze` | 发起冻结 | **需人工确认** |
| `trigger_eod` | 触发日切 | **限定角色** |

**MCP 权限原则**：查询类开放，写入类要确认，管控类限角色。

## 第三章：Prompt 写法详解

### 3.1 四段式标准结构

每一个有效的 Prompt 都包含这四段：

```markdown
## 任务
[一句话说清楚要做什么]

## 输入
[当前任务依赖的最小上下文：表结构、接口定义、方法签名]
[只贴相关的，不要贴整个文件]

## 输出
[期望的产出物：类名、方法签名、文件位置]
[越具体越好]

## 约束
[必须遵守的规则]

## 不要做
[明确禁止的行为——比约束更重要]
```

---

### 3.2 实战示例：从烂到好

**场景**：实现字典管理的「按类型批量查询」接口

**❌ 烂 Prompt：**
```
帮我写字典查询接口
```
> AI 会：自己决定返回结构、自己决定缓存策略、自己决定异常处理方式、
> 可能用 HashMap 而不是项目统一的 Result<T>、可能顺手加上分页、权限校验……

**✅ 好 Prompt：**

````markdown
## 任务
实现字典管理模块的「按类型批量查询」接口，走缓存。

## 输入
表结构（只贴关键字段）：
```sql
t_dictionary(id, dict_type VARCHAR(50), dict_code VARCHAR(50),
             dict_name VARCHAR(100), status TINYINT COMMENT '1启用 2停用')
```
已有基础设施（方法签名）：
- `CacheUtil.get(key)` / `CacheUtil.set(key, value, ttlMinutes)`
- `Result<T>` 统一返回体
- `ServiceException(ResultCode)` 统一异常

## 输出
在 `DictionaryService` 中新增方法：
```java
Result<List<DictionaryVO>> listByType(String dictType)
```
`DictionaryVO` 只包含：`dictCode`, `dictName`

## 约束
- 缓存 key 格式：`dict:type:{dictType}`
- 缓存 TTL：10 分钟
- 只返回 status=1（启用）的字典项
- 缓存 miss 时查 DB，查完回填缓存

## 不要做
- 不要做分页
- 不要加权限校验
- 不要返回 is_system / status 等内部字段
- 不要写 Controller 层，只写 Service 方法
````

---

### 3.3 五个防「AI 自由发挥」技巧

#### 技巧一：用否定句锁死边界

与其说「需要做 A」，不如**同时**说「不要做 B、C、D」：

```markdown
✅ 不要做缓存
✅ 不要做分页
✅ 不要写单元测试
✅ 不要引入新的依赖
✅ 不要写 Controller 层
```

> 原理：否定句直接压低了某些 Token 序列出现的概率（见第一章 1.3）。

#### 技巧二：要求 AI 先复述再执行

在 Prompt 末尾加：

```
在写代码之前，请先用一句话复述你理解的任务目标，确认无误后再开始。
```

这能暴露 90% 的理解偏差，在 AI 生成大量代码之前发现问题。

#### 技巧三：给反例比给正例更有效

```java
// ❌ 不要这样写：
public double calculateAmount() { ... }

// ✅ 要这样写：
public BigDecimal calculateAmount() { ... }
```

> 反例的作用：直接告诉模型「这个 Token 序列概率设为零」。

#### 技巧四：一个会话只做一件事

**错误做法**：在一个对话里让 AI 先写 Service，再写 Controller，再写单测，再优化性能。

**正确做法**：
- 会话 1：写 Service 方法
- 会话 2：（新开对话）基于方法**签名**写 Controller
- 会话 3：（新开对话）基于方法**签名**写单测

> 原理：会话越长，注意力越发散（见第一章 1.2），早期约束影响力持续衰减。

#### 技巧五：长对话后主动重置

当对话超过 10 轮时：

```
请将本次对话的关键结论整理成 3~5 条，我要开启新对话继续。
```

把提炼结果存入 `docs/prompt/context/current-task.md`，下次直接作为新会话的输入。

---

### 3.4 上下文分级注入原则

不是所有内容都要放进 Prompt，按优先级分层注入：

| 优先级 | 内容类型 | 放在哪里 | 示例 |
|--------|---------|---------|------|
| **P0 必须有** | 角色定义 + 技术栈约束 | 规范层（自动注入） | Java 17 / Spring Boot 3.x |
| **P1 本次必须** | 当前任务直接依赖的表结构 / 接口定义 | Prompt「输入」段 | 只贴本次涉及的 3 张表 |
| **P2 按需加** | 上一步的输出作为本步输入 | Prompt「输入」段 | 只传方法签名，不传实现 |
| **P3 不要加** | 已完成模块的完整代码 | ❌ 不要粘贴 | 不要把整个 Service 粘进去 |

**跨步骤只传签名，不传实现：**

```java
// ✅ 正确：只传签名
// AccountService 已有方法：
// Result<AccountVO> getAccount(String accountNo);
// void lockForUpdate(String accountNo); // SELECT FOR UPDATE

// ❌ 错误：粘贴完整实现（浪费窗口，稀释注意力）
@Service
public class AccountServiceImpl implements AccountService {
    // 200 行完整代码...
}
```

## 第四章：AI 规范层设计（工具无关）

> **核心思想**：规范内容和工具无关，先写好规范文本，再按各自工具的方式加载。
> 规范写一次，所有工具都能用。

---

### 4.1 为什么需要规范层

没有规范层，每次对话都要重新告诉 AI 基本规范——这违背了第一章的原理：不变的约束应该放在 System Prompt 层，而不是每次在 User Prompt 里重复。

规范层本质是一段文本，你把它「喂」给 AI 工具，工具在每次对话时自动把它附加到 System Prompt。不同工具的配置方式不同，但规范文本本身完全通用。

---

### 4.2 规范文本：三个核心文件

规范分三层，放在 `docs/ai-rules/` 目录下，**所有工具共用这一份**：

```
docs/ai-rules/
├── general.md          # 通用规范：角色、技术栈、输出格式
├── java.md             # Java 编码规范：分层、异常、事务
├── accounting.md       # 账务领域规范：金额、幂等、并发、禁止行为
└── all-in-one.md       # 以上三文件合并版（供手动粘贴使用）
```

---

#### `general.md`｜通用规范

```markdown
# 通用开发规范

## 角色定义
你是一名资深 Java 金融系统工程师，熟悉复式记账、强一致性事务处理。

## 技术栈
- Java 17 + Spring Boot 3.x
- MyBatis-Plus 3.5.x
- MySQL 8.0（分区表、悲观锁、乐观锁）
- RocketMQ 5.x
- Redis 7.x
- 前端：Vue 3 + TypeScript + Vite + Element Plus + Pinia

## 输出规范
- 所有代码注释使用中文
- 类和方法的 Javadoc 使用中文
- 不生成任何英文注释

## 禁止行为
- 禁止引入项目依赖配置中不存在的新依赖，除非明确被要求
- 禁止使用控制台输出，统一使用日志框架
- 禁止使用魔法数字，必须定义常量或枚举
- 禁止生成任何 TODO 注释占位，要么完整实现，要么明确说明原因
```

---

#### `java.md`｜Java 编码规范

```markdown
# Java 编码规范

## 分层约束
- Controller：只做参数校验和结果返回，不包含任何业务逻辑
- Service：业务逻辑层，不直接操作 HTTP 相关对象
- Mapper：只做数据库操作，不包含任何业务判断
- accounting-api 模块：严禁引入任何持久层依赖

## 异常处理
- 业务异常统一抛出 ServiceException(ResultCode.XXX)
- 禁止空 catch 块
- 禁止将异常吞掉后返回 null

## 事务规范
- 核心记账流程使用编程式事务（TransactionTemplate）
- 禁止在记账相关 Service 方法上使用声明式事务注解
- 事务边界：从锁账户开始，到所有状态字段更新结束

## 统一返回结构
- 所有 Controller 方法返回 Result<T>
- 分页查询统一返回 PageResult<T>
- Service 方法可直接返回业务对象，由 Controller 包装为 Result
```

#### `accounting.md`｜账务领域规范

```markdown
# 账务领域核心约束

## 金额规约
- 所有金额字段类型必须为 BigDecimal，禁止使用 float 或 double
- 金额计算使用 .add() / .subtract() / .multiply()，禁止用 + - * 运算符
- 除法必须指定精度：.divide(x, 6, RoundingMode.HALF_UP)
- 金额比较使用 .compareTo()，禁止使用 == 或 .equals()

## 幂等规约
- 所有写入接口通过 trace_no + trace_seq 联合唯一键实现幂等
- 幂等校验在事务外完成：先查，存在则直接返回原结果，不抛异常

## 账户余额规约（绝对值法则）
- 严禁对余额做负数运算，遵循：new_balance = old_balance ± amount
- 同向相加：new = old + amount
- 反向相减：必须先校验 old >= amount，通过后执行 new = old - amount
- 余额不足：抛 ServiceException(ResultCode.INSUFFICIENT_BALANCE)，不做补偿

## 并发规约
- 实时入账路径：必须使用 SELECT FOR UPDATE（悲观锁）
- 多账户同时加锁：必须按 account_no 升序排序后依次加锁（防死锁）
- 缓冲入账路径：乐观锁优先，失败 3 次后自动升级为悲观锁

## 先证后账原则
- 凭证生成 → 借贷平衡校验 → 持久化 → 过账（顺序不可打乱）
- 借贷不平衡时：阻断流程，不写库，抛出明确异常

## 禁止行为（任何情况下不得违反）
- 禁止直接 UPDATE balance 字段，必须通过过账引擎
- 禁止跳过借贷平衡校验
- 禁止在事务外查询余额、事务内再扣减（ABA 竞态）
- 禁止硬编码任何会计科目编码
- 禁止在 accounting-api 模块引入任何持久层依赖
```

## 第五章：AI API 接口说明

> 理解 AI 工具背后的调用结构，有助于判断「为什么 AI 会这样响应」，
> 也是将来开发 MCP Server（Phase 7）的必要基础。

---

### 5.1 基础请求结构

所有主流 AI 对话接口（OpenAI 兼容格式）的请求结构如下：

```json
POST /v1/chat/completions

{
  "model": "模型名称",
  "temperature": 0.2,
  "max_tokens": 4096,
  "messages": [
    {
      "role": "system",
      "content": "你是一名资深 Java 金融系统工程师..."
    },
    {
      "role": "user",
      "content": "请实现字典查询接口..."
    },
    {
      "role": "assistant",
      "content": "好的，以下是实现..."
    },
    {
      "role": "user",
      "content": "请再加上缓存功能"
    }
  ]
}
```

**字段说明：**

| 字段 | 说明 | 对应我们的概念 |
|------|------|-------------|
| `role: system` | 系统指令，全局约束 | 规范层（Rules） |
| `role: user` | 用户输入 | 我们写的 Prompt |
| `role: assistant` | AI 的历史回复 | 多轮对话历史 |
| `temperature` | 输出随机性，0~1，越低越稳定 | 见第一章 1.4 |
| `max_tokens` | 本次回复最大 Token 数，超过则截断 | — |

**这就是为什么多轮对话会「变长」：** 每次对话都把完整历史作为 `messages` 数组传进去，数组越来越大，总 Token 数越来越接近上下文窗口上限，早期的规范约束影响力越来越弱。

---

### 5.2 响应结构

```json
{
  "id": "chatcmpl-abc123",
  "model": "模型名称",
  "usage": {
    "prompt_tokens": 1024,
    "completion_tokens": 512,
    "total_tokens": 1536
  },
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "以下是实现字典查询接口的代码：\n```java\n..."
      },
      "finish_reason": "stop"
    }
  ]
}
```

**关键字段说明：**

| 字段 | 说明 | 实用意义 |
|------|------|---------|
| `usage.prompt_tokens` | 输入消耗的 Token 数 | 判断上下文是否快满了 |
| `usage.total_tokens` | 总 Token 消耗 | 费用计算依据 |
| `choices[0].finish_reason` | 停止原因 | 见下表 |

**`finish_reason` 取值：**

| 值 | 含义 | 处理方式 |
|----|------|---------|
| `stop` | 正常结束 | 正常处理 |
| `length` | 达到 max_tokens 被截断 | 代码可能不完整，发送「请继续」 |
| `content_filter` | 触发内容过滤 | 调整 Prompt 表述 |

### 5.3 流式输出（Streaming）

AI 编码工具普遍使用流式输出，边生成边显示：

```
// 请求中开启流式
{ "stream": true, ... }

// 响应：一系列 Server-Sent Events，逐 Token 推送
data: {"choices": [{"delta": {"content": "以下"}}]}
data: {"choices": [{"delta": {"content": "是实现"}}]}
data: {"choices": [{"delta": {"content": "字典查询"}}]}
...
data: [DONE]
```

理解流式输出的意义：**AI 是逐 Token 生成的，不是「想好了再写」**。它在生成第 100 个 Token 时，并不知道第 500 个 Token 会是什么。这进一步说明为什么 Prompt 越精确，输出越稳定——精确的 Prompt 在生成初期就锁定了方向。

### 5.4 MCP 协议报文结构

MCP 是在标准 Chat API 之上的扩展，让 AI 能调用外部工具。

**第一步：在请求中定义可用工具**

```json
{
  "messages": [...],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "query_account_balance",
        "description": "查询指定账户的余额，包含可用余额、冻结余额和缓冲预估余额。账户不存在时返回错误。",
        "parameters": {
          "type": "object",
          "properties": {
            "account_no": {
              "type": "string",
              "description": "账户编号，格式：ACC-YYYY-XXXXXX"
            },
            "include_buffer_estimate": {
              "type": "boolean",
              "description": "是否包含缓冲期预估余额（待入账金额），默认 false"
            }
          },
          "required": ["account_no"]
        }
      }
    }
  ],
  "tool_choice": "auto"
}
```

**第二步：AI 决定调用工具时，响应如下**

```json
{
  "choices": [
    {
      "message": {
        "role": "assistant",
        "content": null,
        "tool_calls": [
          {
            "id": "call_abc123",

              "name": "query_account_balance",

          }
        ]
      },
      "finish_reason": "tool_calls"
    }
  ]
}
```

**第三步：你的 MCP Server 执行工具，把结果回传**

```json
{
  "messages": [
    // ... 之前完整的对话历史
    {
      "role": "tool",
      "tool_call_id": "call_abc123",
      "content": "{\"available_balance\": \"9850.000000\", \"frozen_balance\": \"150.000000\", \"buffer_estimate\": \"200.000000\", \"currency\": \"CNY\"}"
    }
  ]
}
```

**第四步：AI 收到结果后给出最终自然语言回复**

```json
{
  "choices": [
    {
      "message": {
        "role": "assistant",
        "content": "账户 ACC-2025-001 当前余额：\n- 可用余额：9,850.00 元\n- 冻结余额：150.00 元\n- 缓冲预估（待入账）：200.00 元"
      },
      "finish_reason": "stop"
    }
  ]
}
```

### 5.5 开发 MCP Server 的关键注意事项

我们项目 Phase 7 会基于上述协议开发 MCP Server，几个关键点提前了解：

**Tool 描述要写清楚边界（AI 靠描述判断何时调用哪个 Tool）：**

```json
// ❌ 描述太模糊
"description": "查询余额"

// ✅ 描述清晰，含输入输出范围和错误情况
"description": "查询指定账户的实时余额。返回可用余额（已过账部分）、冻结余额和缓冲预估余额（待入账）。账户不存在时返回错误码 ACCOUNT_NOT_FOUND。"
```

**写入类 Tool 描述要包含风险提示：**

```json
"description": "向指定账户发起记账操作。⚠️ 此操作直接影响账户余额且不可撤销，执行前请确认参数正确。必须提供 trace_no 作为幂等键防止重复记账。"
```

**参数设计明确必填与选填：**

```json
"required": ["account_no", "trace_no"],  // 必填
// amount 在 properties 中定义但不在 required 里 = 选填
```

**`finish_reason: tool_calls` 是正常响应**，不是错误——它表示 AI 决定调用工具，你需要处理 `tool_calls` 数组，执行对应工具后把结果追加到 `messages` 里继续发请求。

### 5.6 报文膨胀：工具越多、对话越长，单次请求越重

这是最容易被忽视的问题。**每一次 API 调用，都要把当前完整状态全部打包发出去。** 随着工具增加和对话轮次增加，单次请求的报文体积会急剧膨胀。

---

#### 膨胀的三个来源

**来源一：对话历史（多轮累积）**

每轮对话后，`messages` 数组增加两条记录（user + assistant）。10 轮对话后，你发出的请求已经包含 20 条历史消息：

```
第 1 轮请求：messages 数组大小 ≈ 500 tokens
第 5 轮请求：messages 数组大小 ≈ 3,000 tokens（含4轮历史）
第 10 轮请求：messages 数组大小 ≈ 8,000 tokens（含9轮历史）
第 20 轮请求：messages 数组大小 ≈ 20,000+ tokens（含19轮历史）
```

越到后面，新增的内容越少，但每次都要把全部历史重新发一遍。

---

**来源二：工具定义（每次请求都要带上）**

每次请求里都要完整传入所有 `tools` 的定义。工具越多，这部分越大：

```
// 1 个 Tool（简单描述）≈ 150 tokens
// 6 个 Tool（如我们项目规划的）≈ 900 tokens
// 20 个 Tool（复杂参数描述）≈ 3,000+ tokens

// 注意：tools 定义在每次请求都要重复传，不是只传一次
```

我们项目 MCP Server 规划了 6 个 Tool，每次对话请求都会在 `tools` 字段里携带这 900 个 Token，不管 AI 这次用不用这些工具。

---

**来源三：工具执行结果（追加进 messages）**

每次工具调用后，执行结果会以 `role: tool` 的形式追加进 `messages`。如果工具返回的数据量大，报文膨胀更快：

```json
// 工具返回简单余额数据 ≈ 50 tokens，影响较小
{
  "role": "tool",
  "content": "{\"balance\": \"9850.00\"}"
}

// 工具返回完整凭证明细（含多条分录）≈ 500~1000 tokens
{
  "role": "tool",
  "content": "{\"voucher_no\": \"V2025001\", \"entries\": [{...}, {...}, {...}, {...}]}"
}
```

---

#### 三者叠加后的真实规模

一个典型的「对话 10 轮 + 触发 3 次工具调用」场景，单次请求的实际大小：

```
规范层（System Prompt）    ≈  800 tokens   [固定，每次都有]
对话历史（10 轮）          ≈  8,000 tokens  [随轮次线性增长]
工具定义（6 个 Tool）      ≈  900 tokens   [固定，每次都有]
工具执行结果（3 次调用）    ≈  1,500 tokens  [随调用次数增长]
当前 Prompt               ≈  300 tokens
─────────────────────────────────────────
单次请求总输入             ≈  11,500 tokens
```

对比一下「干净的短会话」：

```
规范层                    ≈  800 tokens
工具定义（6 个）           ≈  900 tokens
当前 Prompt               ≈  300 tokens
─────────────────────────────────────────
单次请求总输入             ≈  2,000 tokens
```

**同样的任务，长对话版本的 API 调用成本是短会话版本的约 6 倍。**

---

#### 为什么这很重要

**影响一：费用**

主流 AI API 按 Token 计费。对话越长、工具越多，每次调用花的钱越多。对于高频使用场景（比如 CI 自动化、Agent 批量处理），成本差距会非常显著。

**影响二：响应速度**

输入 Token 越多，AI 处理时间越长，响应延迟越高。对于需要实时交互的场景（比如 MCP Server 的实时查询），长上下文会明显影响体验。

**影响三：质量下降**

见第一章 1.2：上下文越长，注意力越发散，早期约束越容易被「稀释」。报文膨胀不只是成本问题，还直接影响 AI 输出的质量和可预期性。

---

#### 控制报文大小的四个实践

```
实践一：会话要短
  → 超过 10 轮主动开新会话，提炼结论传递，不要带着整段历史继续
  → 一个会话只做一件事（见第三章 3.3 技巧四）

实践二：工具按需注册
  → MCP Server 不要把所有 Tool 都放在一个 Server 里
  → 按场景拆分：查询类一个 Server，写入类一个 Server
  → 当前对话不需要写入操作，就不挂载写入 Server

实践三：工具返回结果要精简
  → Tool 返回值只包含 AI 需要的字段，不要返回完整的数据库记录
  → 例：query_account_balance 只返回三个余额数字，不返回账户的全部字段

实践四：上下文只传签名，不传实现
  → 跨会话传递信息时，只传方法签名，不粘贴完整代码（见第三章 3.4）
  → 把大文档放进 Projects，让工具按需检索，而不是全部塞进 Prompt
```

---

#### 一个直观的类比

把 API 请求想象成每次给 AI 写一封信：

- **对话历史**：你必须把所有往来信件的完整内容抄一遍附进去，因为 AI 没有记忆
- **工具定义**：你必须把所有「实习生能做的事情」清单也附进去
- **工具执行结果**：你还要把「实习生去查了之后带回来的报告」也附进去

**每封信都比上一封更厚，邮费（API 费用）和实习生读信的时间（响应延迟）也随之线性增长。**

保持会话短、工具精简、返回值精炼，就是在控制每封信的厚度。

## 第六章：团队工作模式

### 6.1 时间分配：40-20-40 模型

根据斯坦福 Vibe Coding 课程的建议：

```
传统模式：  10% 规划  →  80% 写代码  →  10% 测试
Vibe Coding：40% 规划  →  20% 执行（AI写）→  40% 验证
```

**具体体现在我们项目中：**

| 阶段 | 占比 | 做什么 |
|------|------|-------|
| **规划（40%）** | 准备 Prompt 的输入段、约束、不要做清单 | 整理表结构、明确边界、设计测试场景 |
| **执行（20%）** | 让 AI 生成代码 | 运行 Prompt，必要时拆成多个会话 |
| **验证（40%）** | Review + 测试 | 逻辑正确性、边界处理、并发安全 |

> **对初级同学的启示**：以前「不会写」是卡点，现在「不会判断对错」才是卡点。
> 提升 Code Review 能力和测试意识，比提升手写代码速度更重要。

### 6.2 系统性学习建议

不建议零散地「用到哪学到哪」，更建议**挑一个完整模块从头跟着做一遍**：

**推荐学习路径（以字典管理模块为例）：**

```

Step 4：不看 AI 的解释，自己逐行 Review，找出至少 3 个可改进点
Step 5：把问题反馈给 AI，让它修复，观察它如何处理
Step 6：自己写单测——测试场景不要让 AI 决定
Step 7：复盘：哪些 Prompt 写法有效？哪些踩坑了？
```
