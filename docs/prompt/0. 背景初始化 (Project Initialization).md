# Role & Context
你现在是一名资深JAVA金融账务架构师。我们将基于已对齐的 DDL 和设计共识开发一套账务核心系统。

本次迭代暂不做 UI，仅提供后端服务与单元测试。

## 一、总体目标

### 实现一个通用账务内核，满足：
- 复式记账（借贷必相等）
- 会计科目树 + 辅助核算
- 内/外部账户统一账户模型（t_account + t_sub_account）
- 凭证规则引擎（t_voucher_rule*）
- 缓冲记账（实时/异步/日终）
- 日余额表 + 余额快照表
- 支持红冲、蓝补、调账

### Logic Baseline (核心逻辑基准)
1. **余额计算**：不在 SQL 中使用 `balance = balance + ?`，必须在 Java 代码层计算。
2. **计算法则**：采用“同向相加、异向相减”。资产类(借方)增加为借，减少为贷；负债类(贷方)增加为贷，减少为借。
3. **并发控制**：实时路径使用 `SELECT ... FOR UPDATE` 悲观锁；缓冲路径使用 `version` 乐观锁。
4. **幂等设计**：分录流水号 `entry_id` 由 `voucher_no + line_no` 拼接，利用数据库唯一索引防重。
5. **红冲逻辑**：取原凭证反向生成，借贷方向取反，金额重新计算。

## 四、需要实现的核心能力（本期）

1. 会计科目与账户
   - 科目树的增删改查（不破坏历史数据）
   - 科目在创建时如果设置了需要设立分户，需要创建内部账户（生成t_account，t_sub_account不需要）
   - 根据 t_account_template 自动开户（生成 t_account + t_sub_account）
2. 记账接口
   - 入参：业务记账请求（trace_no, trans_code, amount, 明细等）
   - 流程：
     1. 保存 t_business_record + t_business_detail
     2. 创建 t_transaction（事务）
     3. 根据 t_voucher_rule* 生成 t_accounting_voucher + t_voucher_entry_detail + t_voucher_auxiliary_detail
     4. 根据记账模式（实时/异步）：
        - 实时：立刻更新 t_account/t_sub_account 和 t_account_detail/t_sub_account_detail
        - 异步：写入 t_buffer_posting_detail，等待任务处理
3. 过账与账户更新
   - 过账控制：凭证状态从“未过账” -> “已过账”，记录 post_time
   - 账户更新：使用乐观锁（version 字段）更新余额和 balance_direction
4. 日终批处理
   - 从 t_account_detail 汇总生成 t_account_balance
   - 从 t_account_balance 或账户表生成 t_account_balance_snapshot
5. 红冲/蓝补/调账
   - 提供接口：对指定凭证生成红字/蓝字凭证，并更新账户、余额表
